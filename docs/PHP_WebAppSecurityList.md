# PHP Webアプリケーション セキュリティ対策一覧 (File I/O & DB I/O 対応)

## 1. データベース入出力（DB I/O）の対策

* **SQLインジェクション対策（最重要）:**
  * 文字列連結によるSQL文の組み立てを一切やめ、必ず**PDO（PHP Data Objects）**または**MySQLi**の「プリペアドステートメント」を使用する。
  * テーブル名やカラム名など、プレースホルダが使えない部分を動的に指定する場合は、ホワイトリスト形式で厳格に検証する。
* **DBユーザー権限の最小化:**
  * Webアプリケーションからデータベースに接続するユーザーには、必要な権限（SELECT, INSERT, UPDATE, DELETEなど）のみを付与し、テーブルの削除（DROP）やシステム権限などは与えない。
* **認証情報の暗号化保存:**
  * パスワードなどをDBに保存する場合は、平文や単純なハッシュ（MD5やSHA1）ではなく、必ずPHP標準の `password_hash()` を使用してソルト付きで安全にハッシュ化する。

## 2. ファイル入出力（File I/O）の対策

* **ディレクトリトラバーサル（パストラバーサル）対策:**
  * 外部からの入力値（GET/POSTパラメータなど）を利用してファイルパスを組み立てる場合、`../` などの文字列を使って意図しないディレクトリへアクセスされるのを防ぐ。
  * 入力値に対して `basename()` を適用してファイル名のみを抽出する、または `realpath()` を使って展開後の絶対パスが許可されたベースディレクトリの配下にあるかを検証する。
* **悪意のあるスクリプトの実行防止（アップロード対策）:**
  * アップロードされたファイルは、極力**ドキュメントルート（Webから直接アクセスできる場所）の外**に保存する。
  * ドキュメントルート内に保存せざるを得ない場合は、そのディレクトリの `.htaccess` やWebサーバーの設定で、PHPなどのスクリプト実行を完全に無効化する（例: `php_flag engine off` や `RemoveHandler .php`）。
* **安全なファイル名の生成:**
  * ユーザーがアップロードした元のファイル名をそのまま保存しない。OSのコマンドインジェクションや文字コード起因の脆弱性を防ぐため、ファイル内容はサーバー側でハッシュ値（SHA-256など）や推測困難なランダム文字列を生成し、それにリネームして保存する。
* **確実なMIMEタイプ・拡張子の検証:**
  * `$_FILES['image']['type']` や単純な拡張子チェックは偽装が可能なため信用しない。PHPの `finfo` クラス（`finfo_file` 関数）を使用して、サーバー側でファイルの実際のシグネチャ（マジックバイト）からMIMEタイプを判定する。
* **ファイルサイズとDoS攻撃対策:**
  * 巨大なファイルを送りつけられてサーバーのディスク容量やメモリを枯渇させられないよう、`php.ini` の `upload_max_filesize` と `post_max_size`、およびメモリー制限 `memory_limit` を適切なサイズに絞る。

## 3. 一般的なWebアプリケーションの対策

* **XSS（クロスサイトスクリプティング）対策:**
  * データベースやファイルから読み込んだデータ、ユーザーの入力値をHTMLとしてブラウザに出力する際は、必ず `htmlspecialchars($string, ENT_QUOTES, 'UTF-8')` を通して特殊文字をエスケープする。
* **CSRF（クロスサイトリクエストフォージェリ）対策:**
  * データの更新やファイルのアップロードを伴うPOSTリクエストには、セッションに紐づいた推測困難なワンタイムトークン（CSRFトークン）を発行し、リクエスト受信時にその整合性を検証する。
* **セッション管理の厳格化:**
  * セッションハイジャックを防ぐため、ログインや権限昇格のタイミングで必ず `session_regenerate_id(true)` を呼び出しセッションIDを再生成する。
  * `php.ini` で `session.cookie_httponly = 1` を設定し、JavaScriptからのセッションクッキー読み取りを禁止する。HTTPS環境であれば `session.cookie_secure = 1` も設定する。

## 4. サーバー・PHP設定レベルの対策

* **エラー情報の隠蔽:**
  * 本番環境では `php.ini` の `display_errors` を `Off` に設定し、ブラウザ上に内部のファイルパスやDBの構造エラーなどが表示されないようにする。エラーの把握は `log_errors = On` にしてサーバー内のログファイルで行う。
* **PHPバージョンの隠蔽:**
  * `php.ini` で `expose_php = Off` に設定し、HTTPレスポンスヘッダ（`X-Powered-By: PHP/...`）からPHPのバージョン情報を消去し、攻撃者に無駄な情報を与えないようにする。
