# JustLinkIt - アーキテクチャ及び仕様設計書

## 1. システム概要
Snipping Toolの自動保存ファイル（画像および動画）を検知し、バックエンドへ自動アップロードした上で、生成された公開URLをクリップボードにコピーする常駐型軽量ツール。さらに「送る」メニューやクリップボードからの手動アップロードにも対応。

## 3. 追加仕様・決定事項

### 3.1. クライアント側の動作設定
以下の機能はユーザーがON/OFFを切り替えられる設計とする。
- **ブラウザ自動表示**: アップロード成功時、取得したURLを既定のブラウザで自動的に開くか。
- **ローカルファイルの削除**: アップロード完了後、PC内の元の画像・動画ファイルを削除するか。

設定変更用の専用ウィンドウ（設定画面）は設けず、**タスクトレイアイコンの右クリックメニューから直接変更する**方式とする（項目数が少ないため）。ON/OFF系はチェック可能なメニュー項目、監視フォルダはフォルダ選択ダイアログ、サーバーURLは簡易入力ボックスで変更し、変更と同時に`settings.json`へ即時保存する。

### 3.2. 動画（MP4）ファイルの取り扱い
Snipping Toolで録画された動画（.mp4）もアップロード対象とする。
- **アップロード形式**: 変換処理は行わず、.mp4のままサーバーへ送信する。
- **閲覧時の挙動**: サーバーは直リンクではなく、**OGPタグ（Open Graph Protocol）を埋め込んだHTMLページ**をレスポンスとして返す。この挙動は**動画のみ**に適用する。
- **目的**: 取得したURLをDiscordやSlack等のツールに貼り付けた際、自動展開されてGIF画像のようにループ再生させるため。
- **画像との違い**: 画像は直リンクのままでもDiscord/Slack等が自動でプレビュー展開できるため、OGPラップを行わず実ファイルへの直リンクを返す（詳細は4章のレスポンス仕様を参照）。

### 3.3. アップロードファイルの命名規則
- **ハッシュ化**: ファイル名の重複防止と元ファイル名の隠蔽のため、アップロードされた**ファイル内容そのもののハッシュ値（SHA-256等）**をファイル名として採用する（URL長は短縮せず、完全なハッシュ文字列を利用する）。

### 3.4. 手動アップロード機能（拡張）
Snipping Toolの監視に加えて、以下のアップロードトリガーを実装する。
- **「送る(Send To)」メニューからの連携**: 
  - エクスプローラーで画像を右クリックし「送る」からアップロード可能にする。
  - **初回起動時のUX**: 勝手に登録せず、初回起動時に「『送る』メニューに追加しますか？」という確認ダイアログを表示する。
- **タスクトレイメニューからの連携**:
  - トレイアイコンの右クリックメニューに「ファイルを選択してアップロード」「クリップボードの画像をアップロード」機能を追加する。

### 3.5. アンインストール（お掃除）機能
レジストリを使用しない軽量アプリだが、不要になった際にSendToのショートカットや設定ファイル等を一括で削除するためのクリーンアップバッチ (`uninstall.bat`) を同梱する。

## 4. API通信仕様

### エンドポイント
- **URL**: `POST /api/upload.php`
- **Content-Type**: `multipart/form-data`

### リクエスト (Request)
| フィールド名 | 型 | 必須 | 説明 |
| :--- | :--- | :--- | :--- |
| `image` | File | Yes | アップロードするファイル本体 (png, jpg, mp4 等) |

### レスポンス (Response)
**成功時 (HTTP 200 OK)**
```json
{
  "success": true,
  "url": "https://yourdomain.com/u/8d969eef6ecad3c29a3a629280e... (ハッシュ値)"
}
```
※ 返却される `url` はファイル種別によって形式が異なる（詳細は3.2参照）。
  - **画像の場合**: 実ファイルへの直リンク（例: `https://yourdomain.com/u/{ハッシュ値}.{拡張子}`）。画像はDiscord/Slack等が直リンクをそのままプレビュー展開できるため、OGPラップは不要。
  - **動画（.mp4）の場合のみ**: OGPタグが設定されたビューアーHTMLへのリンク（例: `https://yourdomain.com/u/{ハッシュ値}`、拡張子なし）。

### 3.6. ギャラリー機能（Webフロントエンド）
アップロードされた画像を閲覧・管理するための、超高速なWebギャラリーを同一サーバー（プロジェクト）内で提供する。

**【UI/UXデザイン要件】**
- **デザイン基調**: ダークモード（黒背景）をベースとした、装飾の少ないソリッドなデザイン。
- **一覧表示**: CSS Gridを用いたカードレイアウト。下部スクロールに応じて追加読み込みを行う「無限スクロール（IntersectionObserverを利用）」を採用する。
- **モーダル表示**: 画像クリック時、画面遷移せずにネイティブの `<dialog>` タグを利用した半透明背景のモーダルを展開し、キーボード（左右キー）で前後の画像へスムーズに遷移できる設計とする。

**【技術スタックとバックエンド連携】**
- **フロントエンド**: React等の重いフレームワークは不使用。ピュアなHTML/CSS/Vanilla JSで構築し、初期レンダリング速度を最優先する。
- **データ取得**: ディレクトリの直接走査（File I/O）ではなく、アップロード時に構築された **SQLiteデータベース (`gallery.sqlite3`)** から画像リストを取得する。これにより、将来的な「複数タグによる絞り込み検索」や、無限スクロール時の**「高速な追加データの取得（チャンク読み込み）」**に対応する。
- **物理削除**: ギャラリー上からの画像削除は、ストレージ容量の節約のため、**「実ファイルの削除」および「SQLiteからのレコード完全削除（物理削除）」**として実装する。
